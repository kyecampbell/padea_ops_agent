## Budget analysis: why cap 15 was too low

The user's T-72h formula `1 + 2×5 = 11` assumed gmail_send counts. Corrections:
- `create_agent_run`/`complete_agent_run` are loop infrastructure, not model-dispatched
  — they do NOT count against the cap.
- `gmail_send` is NOT in TOOL_SCHEMAS in the current build — T-72h per-session count
  is 4 tools (not 5), giving `1 + 2×4 = 9` for the T-72h portion.
- Real cap pressure: quality monitoring. `get_feedback_since` + `compute_rolling_mean`
  × all caterers × 2 windows = 5–9 calls if run naively for all caterers.

Clean 2-session batched run (old design, quality scoped to session caterers): 9 + 5 = 14.
At cap with 1 spare. Any `add_note` → over cap.

Once gmail_send added: `1 + 2×5 = 11` + 5 = 16. Over cap on a clean batched run.

Decision: **per-session runs** (one session per run, see below) + **cap raised to 20**.

Single-session worst case after fix (gmail wired, MOQ shortfall, 2 dietary escalations,
feedback in window, sustained decline detected):
```

get_feedback_since              1

Total:                         13 calls — cap 20 → 7 headroom
```

### Change 2: `agent_context/system_prompt.md`

**Three edits:**

**2a. "What every run does" step 3** — "For each session returned, run the order
composition pipeline" → "Process one session per run — see the order composition
pipeline below for the one-session rule and the five steps."

**2b. "What every run does" step 5** — updated to reflect skip-out: "Call
`get_feedback_since(as_of − 24 hours)`. If the result is empty, skip all remaining
quality steps."

**2c. "Order composition pipeline" section — full replacement** (8 steps → 5 steps):

**New pipeline:**

One-session-per-run rule (at top):
> If `get_sessions_needing_orders` returns multiple sessions, process only the first
> one. `get_sessions_needing_orders` excludes sessions that already have an order —
> the next invocation self-selects remaining sessions via the idempotency filter.
> Each session is a self-contained run with its own decision log entry.

Step 1 — `get_exclusions(school_id, session_date)`: whole-school check only. If school
is physically closed → `add_note(urgency="informational")` and stop.

Step 2 — `compose_session_order(session_slot_id, session_date)` DIRECTLY. One call.
Handles internally: cohort, absences + dietary override, year-level exclusions + dietary
override, meal selection (request → rotation → auto-pick), safety checks, create_order.

**EXPLICIT 6 forbidden internals** — must NOT be called by the model:
- `get_enrolments_for_session`
- `get_absence`
- `get_year_level_exclusions`
- `get_meal_request`
- `get_next_rotation_meal`
- `auto_pick_dietary_meal`

Step 3 — Inspect `result["escalations"]`; `add_note(urgency="urgent")` per entry.

Step 4 — `check_weekly_moq(caterer_id, week_of=session_date)`;
`add_note(urgency="notable")` if shortfall > 0.

Step 5 — `compose_order_email(order_id)` + `gmail_send(...)` when available.

**2d. "Quality monitoring" section — full replacement:**

Since anchor: `since_timestamp = as_of − 24 hours`. Reason: quality is re-evaluated
when recent feedback has arrived; a 24h look-back catches ratings logged since the
last daily run. `as_of` is in the trigger reason.

Early exit: if `get_feedback_since(since_timestamp)` returns empty → stop quality,
no further quality calls this run.

If feedback found: single-session checks for affected sessions; `compute_rolling_mean`
ONLY for caterers that appear in the feedback (not all caterers); sustained-decline
check if threshold crossed → `add_note(urgency="notable")` + draft warning body queued
for approval.
