## Code changes this session

### 1. Bug fix: `src/tools/meals.py` — `get_next_rotation_meal`

Pre-existing bug: `order_lines.created_at` does not exist in schema. The query was:
```sql
SELECT ol.menu_item_id, MAX(ol.created_at) AS last_served
FROM order_lines ol
WHERE ol.enrolment_id = %s
  AND ol.menu_item_id = ANY(%s::bigint[])
GROUP BY ol.menu_item_id
```

Fixed to:
```sql
SELECT ol.menu_item_id, MAX(o.session_date) AS last_served
FROM order_lines ol
JOIN orders o ON o.id = ol.order_id
WHERE ol.enrolment_id = %s
  AND ol.menu_item_id = ANY(%s::bigint[])
GROUP BY ol.menu_item_id
```

The fix is semantically correct — rotation tracks when a student was last served by session date, not row creation time. Bug was hidden because order_lines=0 rows in seed data (PostgreSQL was never executing the query).

**⚠ Watch-item for demo:** `get_next_rotation_meal` uses `MAX(o.session_date)` — correct-by-construction but UNEXERCISED (order_lines=0). Consider seeding a few order_lines before the T-72h demo run to verify rotation visibly works.

### 3. Safety hole closed + safety_records added: `src/tools/orders.py` — `compose_session_order`

**Safety hole identified:** request-source meals never had `item_is_safe_for_enrolment` called. A tutor could file a meal request that violated the student's dietary tags and it would be silently accepted.

**Fix:** Run `item_is_safe_for_enrolment` on ALL three meal-selection paths (request / rotation / auto_pick). For rotation, separated the containment check into `rotation_safe` so `safe` is always explicitly set rather than assumed.

Full rewritten student loop:
```python
order_lines: list[dict] = []
safety_records: list[dict] = []
escalations: list[str] = []

for student in cohort:
    enrolment_id = student["enrolment_id"]
    meal = None
    source = None
    safe = None

    request = get_meal_request(enrolment_id, session_slot_id, session_date)
    if request:
        meal = request
        source = "request"
        consume_meal_request(request["request_id"])
        # Tutor-filed requests name any item on the menu; machine double-check
        # required because the tutor app cannot enforce dietary safety at entry time.
        safe = item_is_safe_for_enrolment(meal["menu_item_id"], enrolment_id)

    if meal is None:
        rotation = get_next_rotation_meal(enrolment_id, caterer_id)
        if rotation:
            rotation_safe = item_is_safe_for_enrolment(rotation["menu_item_id"], enrolment_id)
            if rotation_safe:
                meal = rotation
                source = "rotation"
                safe = rotation_safe  # True; explicit rather than hardcoded — defense in depth

    if meal is None:
        meal = auto_pick_dietary_meal(enrolment_id, caterer_id)
        if meal:
            source = "dietary_auto_pick"
            # auto_pick filters by safety in SQL; re-run check in Python for defense in depth.
            safe = item_is_safe_for_enrolment(meal["menu_item_id"], enrolment_id)
        else:
            escalations.append(
                f"URGENT: no safe meal for {student['student_name']} "
                f"(enrolment {enrolment_id}) — dietary tags "
                f"{student['dietary_tag_names']} — no matching item from caterer {caterer_id}"
            )
            continue

    order_lines.append({
        "enrolment_id": enrolment_id,
        "menu_item_id": meal["menu_item_id"],
        "source":       source,
        "price_cents":  meal["price_cents"],
        "student_name": student["student_name"],
    })
    safety_records.append({
        "enrolment_id":        enrolment_id,
        "student_name":        student["student_name"],
        "menu_item_id":        meal["menu_item_id"],
        "meal_name":           meal["name"],
        "source":              source,
        "dietary_tags":        student["dietary_tag_names"],
        "safe":                safe,
        "other_allergy_notes": student.get("other_allergy_notes"),
    })
```

Return dict now includes `"safety_records": safety_records`.

### Type coercion: `_PARAM_TYPES`

Date/datetime parameters arrive from the Anthropic API as JSON strings and must be coerced before calling the real Python function. A `_PARAM_TYPES` dict maps `(tool_name, param_name) → python_type` for all 12 affected parameters:

```python
_PARAM_TYPES: dict[tuple[str, str], type] = {
    ("get_enrolments_for_session",  "session_date"):  date,
    ("get_meal_request",            "session_date"):  date,
    ("get_absence",                 "absence_date"):  date,
    ("upsert_absence",              "absence_date"):  date,
    ("get_exclusions",              "as_of"):         date,
    ("get_year_level_exclusions",   "as_of"):         date,
    ("get_feedback_for_session",    "session_date"):  date,
    ("get_feedback_since",          "since_date"):    date,
    ("compose_session_order",       "session_date"):  date,
    ("get_sessions_needing_orders", "as_of"):         datetime,
    ("compute_rolling_mean",        "as_of"):         datetime,
    ("project_weekly_cost",         "as_of"):         datetime,
    ("check_weekly_moq",            "as_of"):         datetime,
}
```

Coercion logic in `dispatch()`:
- `date` params: `date.fromisoformat(value)` 
- `datetime` params: `datetime.fromisoformat(value)` then `.replace(tzinfo=BRISBANE)` if naive
- `BRISBANE = ZoneInfo("Australia/Brisbane")`

### Three observation shapes — (b) vs (c) distinction is critical

**(a) Success with data:** plain serialised result dict or list, `is_error` absent.

**(b) "Nothing found" / shortfall / business signal:** `None` serialised as `null` in JSON, or dict with shortfall field, `is_error: False` (absent). Tool ran correctly; data says something. Agent uses business logic (e.g., no absence filed = normal; shortfall > 0 = notify caterer). These are NOT errors.

**(c) Exception / system fault:** `{"error": true, "type": "ExceptionClassName", "message": "..."}`, `is_error: True`. Tool failed to run. Agent uses system-fault escalation path.

**(b) vs (c) is preserved at the Anthropic API boundary via the `is_error` flag.** A `None` from `get_absence` (no absence filed) is `is_error: False`. A `ValueError` from `get_session_slot(999)` (not found) is `is_error: True`. Conflating these is a real bug; the loop explicitly separates them.

### `dispatch()` function

```python
def dispatch(tool_name: str, tool_input: dict) -> tuple[Any, bool]:
    """
    Returns (result, is_error).
    Coerces string→date/datetime, calls the Python function, serialises output.
    Catches all exceptions → (error_obs, True).
    add_note is special-cased: calls log_agent_step inside dispatch, returns None.
    """
```

Serialisation: `_serialise(obj)` handles `date`, `datetime`, `Decimal`, `None`, lists, dicts recursively → JSON-safe types.

### Hard cap

```python
MAX_TOOL_CALLS = settings.max_tool_calls_per_run  # from runtime_config.yaml
call_count = 0
cap_hit = False
```

Before each dispatch block:
```python
if call_count >= MAX_TOOL_CALLS:
    # Log one urgent max_calls_reached step
    log_agent_step(run_id, step_index, urgency="urgent", tool_name="max_calls_reached", ...)
    step_index += 1
    # Feed is_error: True result for every remaining tool_use block in this response
    for block in remaining_tool_use_blocks:
        tool_results.append({"type": "tool_result", "tool_use_id": block.id,
                             "content": json.dumps({"error": True, "message": "cap hit"}),
                             "is_error": True})
    cap_hit = True
    continue  # NOT break — must process all blocks to collect tool_use_ids
```

After block loop, if `cap_hit`:
```python
# ONE uncounted final model call for Claude to summarise
final_response = client.messages.create(...)
call_count stays at MAX (not incremented)
final_text = extract_text(final_response)
complete_agent_run(run_id, f"[CAP HIT at {call_count}/{MAX_TOOL_CALLS}] {final_text}")
return
```

### `add_note` meta-tool

- Dispatched when Claude calls tool named `"add_note"` with `{label, body, urgency}`
- Inside `dispatch()`, calls `log_agent_step(run_id, step_index, urgency=urgency, tool_name=label, tool_input={"body": body}, tool_output_full=None, tool_output_summary=body)`
- The outer loop skips its standard `log_agent_step` call when `tool_name == "add_note"` (dispatch already logged it)
- `step_index` is owned by the outer loop and incremented once per tool_use block — no collision possible
- DB's `UNIQUE (run_id, step_index)` is the hard guard
- `label` (not "add_note") becomes `tool_name` in `agent_steps` — operators see e.g. `"sustained_decline_detected"` not `"add_note"`

### Safety-records post-dispatch rule (loop-owned, never model discretion)

After every `compose_session_order` dispatch, the loop unconditionally iterates `result["safety_records"]` and writes urgent agent_steps rows. This is deterministic — Claude never decides whether to log these.

```python
if tool_name == "compose_session_order" and not is_error:
    for rec in result.get("safety_records", []):
        if rec["safe"] is False:
            log_agent_step(
                run_id, step_index,
                urgency="urgent",
                tool_name="dietary_safety_violation",
                tool_input={
                    "enrolment_id": rec["enrolment_id"],
                    "student_name": rec["student_name"],
                    "menu_item_id": rec["menu_item_id"],
                    "meal_name":    rec["meal_name"],
                    "dietary_tags": rec["dietary_tags"],
                },
                tool_output_full=None,
                tool_output_summary=(
                    f"UNSAFE: {rec['student_name']} assigned {rec['meal_name']} "
                    f"but dietary tags {rec['dietary_tags']} not fully covered"
                ),
            )
            step_index += 1
        if rec.get("other_allergy_notes"):
            log_agent_step(
                run_id, step_index,
                urgency="urgent",
                tool_name="unverified_allergy_note",
                tool_input={
                    "enrolment_id":        rec["enrolment_id"],
                    "student_name":        rec["student_name"],
                    "other_allergy_notes": rec["other_allergy_notes"],
                },
                tool_output_full=None,
                tool_output_summary=(
                    f"ALLERGY NOTE (unverified): {rec['student_name']} — "
                    f"{rec['other_allergy_notes']}"
                ),
            )
            step_index += 1
```

Two urgent reasons are DISTINGUISHABLE:
- `"dietary_safety_violation"` = confirmed dietary violation (item.tags ⊄ student.tags, machine-checkable)
- `"unverified_allergy_note"` = requires human verification — free-text allergy, not machine-checkable
