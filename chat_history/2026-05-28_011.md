### caterers.py — built and approved

**Declarations (all three read-only, no `conn.commit()`):**

#### 1. Distance model — `caterers_within_range`
V4 schema has postcodes only — no lat/lon, no stored distance. `pgeocode` not installed; `geopy >= 2.4.0` is installed. Implementation: hardcoded dict of the 8 specific QLD demo postcodes → `(lat, lon)` centroid, plus `geopy.distance.geodesic().km` (Vincenty ellipsoid). Measured school postcode centroid → caterer home_postcode centroid. Compared against `caterers.max_delivery_km`. Raises `ValueError` on unrecognised postcode (fails loud, never silent).

Tested against ISHS (4068): Lakehouse=28.89 km, Terrific=5.21 km, GyG=6.99 km, Kenko=0.0 km (same postcode). All within 50 km. Exclusion filter works.

#### 2. `project_weekly_cost` — integer cents + zero margin
- Tables read: `enrolments` (cohort_size), `menu_items` (avg price), `session_slots` (sessions/week), `caterers` (delivery fee, moq_5_items)
- `per_meal_price_cents = ROUND(AVG(price_cents))::int` — one round at the DB, all subsequent arithmetic is int × int
- V3-MP-01 zero margin: projected_total = caterer price × meals + delivery, no Padea markup
- MOQ: variety_count=5 assumed for projection (middle tier, typical full-rotation). Uses `moq_5_items`. `moq_floor_cents = None` if not triggered; an int if shortfall

Tested Terrific/JPC: cohort=63, price=2050c, sessions=2, projected_items=126 ≥ moq_5=20 → `moq_floor_cents=None`, `projected_total_cents=261300` (=126×2050+3000). Exact value asserted from independently computed formula.

#### 3. `check_weekly_moq` — shortfall returned as data
- Week = ISO Monday–Sunday: `week_start = week_of - timedelta(days=week_of.weekday())`
- `variety_count = COUNT(DISTINCT order_lines.menu_item_id)` for the caterer+week
- Applicable tier: 4→`moq_4_items`, 5→`moq_5_items`, 6→`moq_6_items`, else `None`
- `shortfall = max(0, moq_applicable - total_items)` if applicable, else 0
- `shortfall_cents = shortfall × avg_price_of_ordered_items` (falls back to caterer avg active price if no items)
- Never swallows: always returns `{total_items, moq_applicable, shortfall, shortfall_cents}`

Tested with Terrific, test week 2026-06-22:
- Shortfall case: 10 lines, 5 varieties → shortfall=10, shortfall_cents=20500 (10×2050) ✓
- Met-minimum: 25 lines, 5 varieties → shortfall=0, shortfall_cents=0 ✓
- Test data inserted and cleaned up in `finally` (CASCADE deletes order_lines) ✓

## REVISION NOTE — caterers.py (same session, post-review)

**Status: APPROVED**

### Issues resolved

1. **GST missing from money path** — `project_weekly_cost` and `check_weekly_moq` both lacked GST normalisation. Seed has mixed GST basis: Lakehouse + Terrific are GST-exclusive (`price_includes_gst=False`); Kenko + GyG are GST-inclusive. Decision: normalise ALL prices to GST-inclusive basis before arithmetic. Fetched `price_includes_gst, gst_rate_percent` from caterers in both functions; applied `round(price × (1 + rate/100))` when `price_includes_gst=False`. Returns now represent what Padea actually pays. "Zero margin" (V3-MP-01) means no Padea markup — it says nothing about GST.

2. **Delivery fee applied once per week, not per session** — `project_weekly_cost` was adding `delivery_fee_cents` once regardless of session count. Schema describes it as a "per-delivery flat fee"; caterer delivers to each session. Fix: `delivery_fee_cents * sessions_per_week`. Matters for multi-session schools (ISHS = 3 deliveries/week would have been undercounted 3×, feeding bad RFP comparisons).

3. **`exclude_caterer_id` semantics clarified** — explicitly documented in docstring: this is an RFP incumbent exclusion (caterer omitted from candidate list entirely before distance check runs), not a distance filter.

4. **Postcode dict coverage confirmed** — distinct postcodes across schools (4165, 4127, 4109, 4068, 4151, 4170) and caterers (4165, 4101, 4068, 4000) = 8 unique values. All 8 present in `_QLD_POSTCODES`. No gaps; no real query will throw mid-demo.

5. **Geodesic label corrected** — module docstring and `_distance_km` docstring changed from "Vincenty" to "Karney's geodesic" (geopy's actual algorithm).

### Final test values (Terrific/JPC, all GST-inclusive)
- `per_meal_price_cents`: 2050 (raw) → **2255** (round(2050 × 1.1))
- `delivery_fee_cents`: 3000 (raw) → **3300** (round(3000 × 1.1), per delivery)
- `projected_total_cents`: **290730** (126 × 2255 + 3300 × 2 sessions, moq_floor=None)
- `shortfall_cents` (10-item shortfall case): 20500 → **22550** (10 × 2255)

All 4 tests pass.
