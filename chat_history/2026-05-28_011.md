### compose_order_email — revised and approved

**Function:** `compose_order_email(order_id: int) -> str`
**Location:** `src/tools/orders.py`
**Status:** read-only, no `conn.commit()`

Three issues from initial review resolved before approval:

#### 1. Dietary safety check (option b — containment flag)
The per-student allocation line now:
- Fetches the student's dietary requirements (via `enrolment_dietary_tags → dietary_tags`)
- Fetches the assigned meal's tags (via `menu_item_dietary_tags → dietary_tags`) using correlated subqueries to avoid a Cartesian product from double LEFT JOIN
- Computes containment: `unsafe = bool(student_set - meal_set)` — the meal must be a superset of the student's tags
- Flags `⚠ UNSAFE MATCH` when the meal does NOT cover the student's requirements
- Surfaces `other_allergy_notes` as `⚠ ALLERGY NOTE (unverified): {text}` — explicitly marked unverified because free-text cannot be auto-checked against tags

#### 2. Delivery time
`"Meals start: {dinner_str}"` replaced with:
```
Deliver by: {dinner_time − 10 min} (dinner at {dinner_time})
```
e.g. `Deliver by: 17:50 (dinner at 18:00)`

#### 3. Location deduplication
`schools.building` and `session_slots.room` are separate schema fields. All 11 seeded slots have identical values (e.g. "G Centre" / "G Centre"), producing "G Centre, G Centre". Fix: if `building == room`, render once:
```python
if building and room and building != room:
    location = f"{school_name} — {building}, {room}"
elif room:
    location = f"{school_name} — {room}"
elif building:
    location = f"{school_name} — {building}"
else:
    location = school_name
```

**Light-coverage note (do not re-raise as a bug):** The unequal case (`building != room`, e.g. "Main Building" / "Room 12") is correct by construction but untested — all seed slots are equal. Log as a known light-coverage spot if a coverage report is ever needed.

## Rulings locked this session

### (a) V4 summary wins over tools_reference on document conflicts
**Rule:** When `docs/v4_optimised/v4_summary.md` and `agent_context/tools_reference.md` contradict each other, V4 summary is authoritative for design decisions. tools_reference is the build spec for function signatures and call semantics, but it does not override V4 design intent.

**Applied here:** tools_reference listed "predicted total cost with delivery fee" in compose_order_email's body spec. V4 summary says explicitly "No costs appear on the per-session order. … Pricing and payment totals are reserved for the Monday consolidated summary." V4 summary wins — `compose_order_email` renders a cost-free delivery brief.

**Stale tools_reference line to reconcile later:** `compose_order_email` body spec in tools_reference.md still lists "per-meal totals, predicted total cost with delivery fee" — should be updated to match V4 design before loop.py is built.

### (b) orders.total_cost_cents is the authoritative total
**Rule:** `orders.total_cost_cents` is the authoritative GST- and MOQ-floor-inclusive predicted total. Anywhere costs need to be shown (Monday consolidated summary, etc.), READ this column — never recompute from order_lines.

**Rationale:** The stored value captures what was known at composition time (including any MOQ floor adjustments). Recomputing from order_lines would silently drop the MOQ variance and could diverge if prices change after composition.
