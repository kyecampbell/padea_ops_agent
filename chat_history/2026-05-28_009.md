### infrastructure.py — APPROVED

**File:** `src/tools/infrastructure.py`

**Functions built:**
| Function | Writes? | Notes |
|---|---|---|
| `create_agent_run(trigger_reason)` | ✅ | Inserts into `agent_runs`, returns `run_id` |
| `complete_agent_run(run_id, notes)` | ✅ | Sets `completed_at = now()` + notes |
| `log_agent_step(run_id, step_index, tool_name, tool_input, tool_output_full, reasoning, urgency)` | ✅ | Inserts into `agent_steps`, casts to `step_urgency` enum, uses `psycopg.types.json.Jsonb` for jsonb columns |
| `check_and_resolve_crashed_runs(staleness_minutes=30)` | ✅ | UPDATE WHERE completed_at IS NULL AND started_at < now() - interval; returns list of closed run_ids |

**Key decisions:**
- V4 has no `status` column on `agent_runs`. "Crashed" = stale `completed_at IS NULL`. The fix closes stale runs by setting `completed_at = now()` with a `"CRASHED: ..."` prefix in notes. Consequence: crashed runs are only distinguishable by that free-text prefix — logged as a known V4 limitation (V5 trigger if programmatic query of crashed runs becomes needed).
- Crash note built in Python as an f-string, not via SQL string concatenation (`|| %s ||`). Postgres has no implicit `text || integer` cast — would throw at runtime.
- `Jsonb()` wrapper used for `tool_input` and `tool_output_full` columns (psycopg3 pattern for jsonb).

**Smoke test result:** All 4 functions green. `check_and_resolve_crashed_runs` correctly skipped the just-completed run on second call.

### sessions.py — APPROVED (after one fix)

**File:** `src/tools/sessions.py`

**Functions built:**
| Function | Writes? | Notes |
|---|---|---|
| `get_sessions_needing_orders(as_of, window_hours=1.0)` | ❌ | Read-only |
| `get_session_slot(session_slot_id)` | ❌ | Read-only |

**`get_sessions_needing_orders` algorithm:**
- T-72h window: `target_low = as_of + H - window_hours`, `target_high = as_of + H + window_hours`
- For each active session_slot, scans dates in range(-1, 3) around `target_low.date()` for matching `day_of_week`
- Builds `session_dt = datetime.combine(candidate_date, start_time, tzinfo=BRISBANE)` and tests against window
- Excludes `(slot, date)` pairs that already have an order
- Returns `{session_slot_id, session_date, school_id, caterer_id, session_start_datetime}`
- **Precondition comment added:** `-1..+2 scan only guarantees one match per slot when window_hours << 24h. Do not call with window_hours >= 24.`

**Timezone:** `zoneinfo.ZoneInfo("Australia/Brisbane")` — UTC+10, no DST, fixed offset. Naive `as_of` treated as Brisbane local.

**Key fix applied this session:**
- Original had `WHERE sc.current_caterer_id IS NOT NULL` — silently dropped mid-RFP schools. Removed. Null `caterer_id` now surfaces to the agent layer for escalation. V4 principle: gaps surface as escalations, never silent drops.

**Anchor confirmed:** T-72h is measured from `start_time` (not `dinner_time`). V4 summary: "A Monday session gets its order at Friday 3 PM" — session `start_time` ~15:30, order at 15:30 Friday = exactly 72h before. `dinner_time` (~17:30) would fire 2h early.

**Smoke test result:** All 11 active slots returned at their exact T−72h mark. Already-ordered historical slot correctly excluded. ValueError on missing slot_id confirmed.

```
UNIQUE (enrolment_id, absence_date)
```

```
id bigserial PK | school_id bigint FK (nullable) | enrolment_id bigint FK (nullable)
```
