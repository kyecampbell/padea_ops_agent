**Key behaviours:**

**Corrections locked in 3a:**

1. **`reasoning=None` hardcoded — WAS WRONG, FIXED.** The 3a submission had
   `reasoning=None` on every `log_agent_step` call. Claude routinely emits a
   text block alongside tool_use blocks; that text is the model's stated
   reasoning for those calls and the whole point of the decision log. Fixed by
   extracting `turn_reasoning` once per response turn and passing it on each
   `log_agent_step` call.

   ```python
   turn_reasoning: str | None = next(
       (b.text for b in response.content if b.type == "text"), None
   )
   ```

   `turn_reasoning` is per-response (one string shared across all tool_use blocks
   in the same turn). None when the model goes straight to tool calls with no
   preamble — correct.

2. **`step_index += 1` placement — corrected from spec pseudocode.** Session 015
   pseudocode had `step_index += 1` inside `if tool_name != "add_note"`. For
   `add_note`, dispatch logs the step at `step_index`; if the loop then did NOT
   increment, the next non-add_note call would attempt to log at the same
   `step_index` → UNIQUE (run_id, step_index) constraint violation. Moved
   outside the `if` block: always increments once per tool_use block regardless
   of type. Confirmed in the test: add_note + any subsequent block no longer
   collide.

3. **Items confirmed correct in 3a (no change needed):**
   - Assistant turn: `messages.append({"role": "assistant", "content":
     response.content})` appended BEFORE the tool_results user turn (lines
     948–949 in final file). The API requires user/assistant alternation and
     tool_result must immediately follow the turn containing the matching
     tool_use.
   - One tool_result per block: the for loop iterates every block in
     `tool_use_blocks` and appends one entry keyed by `block.id`. All sent in
     a single user turn after the loop.
   - `max_tokens=4096` — confirmed real value.

## STEP 3b — APPROVED

**What was built:** Hard-cap logic integrated into `run()`.

**New variables before try block:**
```python
call_count = 0                          # cumulative dispatches this run
MAX = settings.max_tool_calls_per_run   # 15 in production
```

**New per-turn variable (inside while loop, before for loop):**
```python
cap_hit = False  # reset each response turn
```

**Cap check — verbatim, placed before dispatch inside the for-block loop:**

```python
for block in tool_use_blocks:
    tool_name = block.name
    tool_input = block.input

    # ── HARD CAP CHECK — before dispatch ─────────────────────────
    if call_count >= MAX:
        if not cap_hit:
            log_agent_step(
                run_id=run_id,
                step_index=step_index,
                tool_name="max_calls_reached",
                tool_input={
                    "blocked_tool": tool_name,
                    "call_count": call_count,
                    "max": MAX,
                },
                tool_output_full=None,
                reasoning=(
                    f"Hard cap {MAX} reached; aborting remaining "
                    f"tool calls in this response"
                ),
                urgency="urgent",
            )
            step_index += 1
            cap_hit = True
        tool_results.append({
            "type": "tool_result",
            "tool_use_id": block.id,
            "content": json.dumps({
                "error": True,
                "type": "HardCap",
                "message": f"Max tool calls ({MAX}) reached",
            }),
            "is_error": True,
        })
        continue

    # ── dispatch ──────────────────────────────────────────────────
    result, is_error = dispatch(tool_name, tool_input, run_id, step_index)
    call_count += 1

    if tool_name != "add_note":
        log_agent_step(
            run_id=run_id,
            step_index=step_index,
            tool_name=tool_name,
            tool_input=tool_input,
            tool_output_full=(
                result if isinstance(result, dict) else {"value": result}
            ),
            reasoning=turn_reasoning,
            urgency="urgent" if is_error else "none",
        )

    step_index += 1

    tool_results.append({
        "type": "tool_result",
        "tool_use_id": block.id,
        "content": json.dumps(result),
        "is_error": is_error,
    })
```

**Final summarising call — after for loop, inside while loop:**

```python
messages.append({"role": "assistant", "content": response.content})
messages.append({"role": "user", "content": tool_results})

if cap_hit:
    # One UNCOUNTED final call — no tools so the model is forced to
    # respond with text only (it cannot recurse into more tool calls).
    # This call is not counted toward call_count and does not loop.
    final = client.messages.create(
        model=settings.agent_model,
        max_tokens=1024,
        system=system_prompt,
        messages=messages,
    )
    final_text = next(
        (b.text for b in final.content if b.type == "text"), ""
    )
    complete_agent_run(
        run_id, f"[CAP HIT at {call_count}/{MAX}] {final_text}"
    )
    return
```

**Design decisions locked in 3b:**

1. `cap_hit` flag: log the `max_calls_reached` step ONCE (first block that trips
   the cap). All subsequent cap-tripped blocks in the same response get
   is_error'd silently — no repeat steps. `continue` (not `break`) so the API
   receives a tool_result for every tool_use block it emitted.

2. Final call has **no `tools` parameter**. Without tools, the Anthropic API
   cannot give the model tool_use capability — the model can only respond with
   text. This is an API-level guarantee against recursion, not just a code-level
   one. Confirmed in test: `final call kwargs keys: ['model', 'max_tokens',
   'system', 'messages']`.

3. `call_count` persists across while-loop iterations (declared before the try
   block). The cap applies to the total dispatches across all turns of the run,
   not just one response.

4. `cap_hit = False` is reset per while-loop iteration. In practice this is
   structural — once `cap_hit → return`, there is no next iteration. But the
   reset is correct: if somehow cap_hit were carried over, a non-cap turn that
   happened to follow would incorrectly fire the final call.

**What to build:** Safety_records unconditional logging wired after every
`compose_session_order` dispatch.

**Full spec (locked — do not relitigate):**

After any `compose_session_order` dispatch that returns `is_error=False` and a
dict result, the loop UNCONDITIONALLY iterates `result.get("safety_records", [])`:

```python
# SAFETY RECORDS RULE — non-negotiable, after every compose_session_order
if tool_name == "compose_session_order" and not is_error and isinstance(result, dict):
    for rec in result.get("safety_records", []):
        if rec.get("safe") is False:
            log_agent_step(
                run_id=run_id,
                step_index=step_index,
                tool_name="dietary_safety_violation",
                tool_input={"enrolment_id": rec["enrolment_id"],
                            "menu_item_id": rec["menu_item_id"]},
                tool_output_full=rec,
                reasoning=(
                    f"UNSAFE MATCH: {rec['student_name']} assigned "
                    f"{rec['meal_name']} but student tags "
                    f"{rec['dietary_tags']} not covered by meal tags"
                ),
                urgency="urgent",
            )
            step_index += 1
        if rec.get("other_allergy_notes"):
            log_agent_step(
                run_id=run_id,
                step_index=step_index,
                tool_name="unverified_allergy_note",
                tool_input={"enrolment_id": rec["enrolment_id"]},
                tool_output_full={"other_allergy_notes": rec["other_allergy_notes"],
                                  "student_name": rec["student_name"]},
                reasoning=(
                    f"Unverified allergy note for {rec['student_name']}: "
                    f"{rec['other_allergy_notes']} — requires human verification"
                ),
                urgency="urgent",
            )
            step_index += 1
```

**Placement:** Inside the for-block loop, AFTER the normal `log_agent_step` and
`step_index += 1` for `compose_session_order`, before appending the tool_result.

**Key rules:**
- NEVER at model discretion — this logging fires deterministically in the loop
  regardless of what the model does next
- `safe=False` → `dietary_safety_violation` urgent step
- Non-null `other_allergy_notes` → `unverified_allergy_note` urgent step
- Both checks are independent (a record can trigger both)
- Uses `reasoning` field (confirmed: `log_agent_step` real parameter is
  `reasoning`, not `tool_output_summary`)
- Loop-owned `step_index` increments for each safety step

## FORWARD FLAG — HTML decision-log renderer

When Claude fires N tool calls behind one text block, all N steps log the same
`turn_reasoning` string. This is correct at the loop layer. The HTML renderer
should group or dedupe steps that share a turn's reasoning so it reads as "one
decision → N tool calls" rather than N identical reasoning cards
(copy-paste artifact in the UI). Note this in the renderer build and mention
as grouped reasoning in the demo.
